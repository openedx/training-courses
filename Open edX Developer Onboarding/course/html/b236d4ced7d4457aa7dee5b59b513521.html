When working in any codebase, you will inevitably have questions about how to write a specific line of code. Open source communities thrive on helping each other solve problems; the question is simply, where should you go with questions while developing? It depends on the type of question that you have.

If you have questions about how a feature should actually work, the design of the feature, or other Product/UX questions, you should engage with the product team on the feature request or roadmap ticket for the feature. There are various Product Working Group Meetings you can attend and ask questions about things like LTI integrations and UX/UI design. Get in touch with the product team early and often as you are implementing your feature so you can be assured your product will be accepted with minimal revisions when it comes time to submit your code. If you’re not sure which Product group to reach out to, start with the Core Product Working Group - they’ll be able to help you out.

Have questions about the actual implementation of a feature or bugfix? The Forums are your best place for advice. Before posting, please search the forums - it’s possible your question has already been asked. When posting, be sure to pick a proper category for your questions, so you alert the right people. The “Development” category, and its subcategories, are usually the right ones to choose to ask about development questions. Other useful categories include the “Site Operators” category for questions about running your own Open edX instance (as described in Module 3), or the “Educators” category for questions about how to author a course (including how to use various problem types).

You can also open GitHub Issues in repos you’re working in to file bug reports and engage in conversations with project maintainers (OEP-55, Project Maintainers defines maintainership). You can find the named maintainers by viewing the “owner” field in the top-level catalog-info.yaml file in each repo (for example, see the one in the open-edx-proposals repo).
