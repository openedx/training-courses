Internationalization (i18n)

The only way that a site can be translated is if its strings are marked appropriately for translation; this is called internationalization (or “i18n”: “i” + 18 letters + “n”). There is special syntax in Python/Django (`gettext`, aliased to `_`) and JavaScript/React (react-intl library) to define strings in a way that tells a scraping script to scrape the strings into a translation file. Translators translate these strings, the translated strings are returned to translation files, and software serves strings according to user preference. How does this work?

First off, “i18n” actually refers only to the act of defining your strings for eventual translation. The process after internationalizing your strings is called localization, or l10n.

To begin the l10n process, strings are scraped from Python/Django files and from React projects using native tooling. For example, here is a description of the Django tooling. From there, OEP-58 defines how we manage translations in the project. Translation files live in their own repo, and a CLI tool (called Atlas) moves translation files from the repository that contains them to where they are needed for development or deployment. This is to preserve Git history & repo conciseness; prior to the Atlas system, translations were checked in weekly, causing a cluttered Git history as well as a bloated repo size.
