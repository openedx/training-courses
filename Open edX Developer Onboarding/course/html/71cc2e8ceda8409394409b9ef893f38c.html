One of the strengths of an open source codebase lies in the vast investment of human resources in reviewing and testing the codebase. With so many “eyes” on the system, problems can be quickly identified and addressed. Following a few simple protocols ensures the safety and security of the software in all of its instances.

If you find a security issue (something that compromises the integrity of Open edX instances or the personal data of Open edX users), please do not report it publicly. Write a report with as much information as you can, and email it to security@openedx.org. See our project security policy for more detail.

If you find a bug, you should first write up a bug report in the repo closest to the bug. A good bug report effectively communicates the steps to reproduce the bug and the expected behavior that should have occurred. If you’re not sure where to report a bug, you can ask advice in the forums or Slack, or report it in the public-engineering project.

This article from Marker.io gives more in-depth tips about writing good bug reports.

When a security issue is found, a post will be made in the Security Announcements section of the forums around 48 hours before the fix is released. When a fix is released, it can be found on the Security Advisories section of the openedx GitHub organization.
