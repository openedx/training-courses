Before we get to the next module, focused on writing code, let’s review the basic standard the Open edX community has established for our codebase. As a general rule, we prefer code that is verbose and easy to understand. Yes, there are times you can do something complicated in a nested list comprehension, but please unfold it into loops that future developers can read and understand. Utilize variables with meaningful names, i.e., course_key rather than ck. Docstrings should be provided that explain both what a function does and what its arguments are. Add enough comments to explain any complicated logic or unfortunate hacks.

In addition to the above, we utilize some published guidelines for each of the coding languages:

Python style linters

    • We use the pep8 tool to follow PEP-8 guidelines.
    • We use pylint for static analysis and to uncover trouble spots in our code.
    • See also: Python style guide

JavaScript style linter

    • We utilize ESLint, a static code analysis tool.
