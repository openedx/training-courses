As mentioned in the last unit, MFEs interact with edx-platform through APIs. MFEs don’t have a database; all the data is supplied to MFEs through the APIs. These APIs are used to read, write, update, and delete data in edx-platform. MFEs don’t have their own open APIs - you cannot make an API call to an MFE. MFE APIs are triggered by user interaction (either the learner, instructor or superuser).

If you are interested in working with MFEs, you will want to familiarize yourself with the documentation for the LMS APIs . Other Django services may have additional APIs, which you can view on your local machine. Most Django services have a live API view which you can reach by running the service locally and going to its url + /api-docs as shown here:

You should also review this reference for API best practices.

A frontend API (an “external” API) is designed to assume that the user upon whom the action is being taken is the user who is making the request. For example, the name_change endpoint is a POST to /api/user/v1/accounts/name_change/ and will change the profile name of the user making the call.

However, an API made for internal systems (such as this second name_change endpoint, POST /api/user/v1/account/name_change/{username}/confirm) will ask for the requester to supply username or user id of the user to update. Any superuser can make this change.

How do you know if an API is designed for external or internal usage? Someone may have documented that intention in the API’s docstring (which you can see in the generated API docs), but in many cases we haven’t done this well, so you’ll need to inspect code. Unfortunately, it can be hard to glean intention automatically from the name or structure of the API. Searching and/or posting to the forums can be helpful if you get stuck.

An example of how APIs interact can be seen in courseApiService.js. Here an API call is made to <LMS_BASE_URL>/api/courses/v1/courses/ in courseApiService.js which will get a list of courses from edx-platform to the MFE.

A more functional example can be seen in StudioApiService.js. This demonstrates the use of LMS/Studio APIs in MFEs. In StudioApiService.js API call is made to STUDIO_BASE_URL/api/contentstore/v1/proctored_exam_settings/<courseID>. It will get a proctored exam setting for a course with course_id. This information is supplied to the MFE.
