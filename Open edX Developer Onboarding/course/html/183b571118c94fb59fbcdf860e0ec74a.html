One of the newest features added to the Open edX platform, MFEs have greatly enhanced the system’s move to being more lean and agile. MFEs stand for Micro Frontends. What are micro frontends? MFEs are small bits of frontend code, as compared with frontend monoliths (usually one code repo holding all the code for a given project). MFEs live in their own repo and are deployed independently. Why use them?

Some of the key benefits that we've seen from micro frontends are:

      • smaller, more cohesive and maintainable codebases
      • more scalable organizations with decoupled, autonomous teams
      • the ability to upgrade, update, or even rewrite parts of the frontend in a more incremental fashion than was previously possible

Credit: This blog post by Martin Fowler, which is worth a read. It gives a very clear idea about what Micro Frontends are.

MFEs help us to scale Frontends and maintain them. There can be multiple small teams that can work on and maintain these different frontends in an agile fashion: without the monolith, teams can develop a feature without worrying about major conflicts.

MFEs interact with edx-platform through APIs. MFEs don’t have a database; all the data is supplied to MFEs through the APIs. These APIs are used to read, write, update, and delete data in edx-platform. MFEs with edx-platform work like the Hub and Spoke model, where edx-platform is the hub and each of the MFEs are spokes.

There are different applications like frontend-app-learning which serves the LMS frontend, and frontend-app-discussions, which serves the frontend for the Open edX native forums.
