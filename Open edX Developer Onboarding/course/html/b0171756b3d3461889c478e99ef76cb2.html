Now, what if we want to explore the libraries installed in the Open edX service, contribute to the development of an existing library, or even we want to turn on our favorite intellisense tool for our mounted volume â€“ like edx-platform? Then, we need the virtual environment used by the service. Try this:

tutor dev copyfrom lms /openedx/venv ./venv-openedx
tutor mounts add lms,cms,lms-worker,cms-worker:./venv-openedx:/openedx/venv
tutor mounts list
- name: lms,cms,lms-worker,cms-worker:./venv-openedx:/openedx/venv
build_mounts: []
compose_mounts:
- service: lms
container_path: /openedx/venv
- service: cms
container_path: /openedx/venv
- service: lms-worker
container_path: /openedx/venv
- service: cms-worker
container_path: /openedx/venv
tutor dev start -d

You can now modify your library after starting your services:

code venv-openedx/lib/python3.8/site-packages/openedx_events

Modifying your library is necessary when you want to contribute to an existing library. For example, if you want to add a new definition to openedx-events, you can install it directly in the venv as shown here. Developing a new XBlock also requires installation in the venv.
