Writing a Roadmap Issue or Feature Request

Writing up a roadmap issue (functionality your team is planning to work on) or feature request (a feature you’d like to see, but are not currently planning to work on) seems easy at first, but it can be kind of tricky. For example, how do you write (for the purposes of communicating with somebody else who is not you, the person writing it right now) a feature ticket including acceptance criteria, technical requirements, and functional requirements?

We’ve tried to simplify the process of opening a roadmap issue or feature request by implementing Roadmap Issue & Feature Request Templates. When submitting a ticket, please fill out these fields with as much detail as you possibly can. After submitting, follow up in the comments section with any relevant screenshots, drawings, or mock-ups that we can use to understand your request. Again, you should read the full instructions for how to submit an open source contribution for Product Review for up-to-date information on the process.

Here are some examples of well-written tickets:

Roadmap issue: https://github.com/openedx/platform-roadmap/issues/278

      • This ticket provides a clear abstract, that is, 1-3 sentences that concisely describe the change.
      • Further, it describes two possible implementation approaches, and the positives and negatives of each approach, giving the technical reviewers a good understanding of the problem right away.
      • It provides a link to a public Google document with additional, detailed information.
      • The author is very responsive to questions within the comments.

Roadmap issue: https://github.com/openedx/platform-roadmap/issues/255#issuecomment-1571088012

      • Like the above, there is plenty of detail in the abstract, as well as the other template sections.
      • This ticket very clearly describes three features (dashboards and metrics) that will be added.
      • There are multiple public links to more in-depth documentation as the project progresses.

Roadmap issue: https://github.com/openedx/platform-roadmap/issues/355

      • The language in this proposal is clear and concise. It is easy to understand what problem the author is seeking to address, and how they plan to do it.
      • They provide a link to a detailed, public product proposal document that includes screenshots of their proposed design.
      • Once again, the author is quite responsive on both the GitHub ticket and the product proposal document.
