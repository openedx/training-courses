The Open edX platform is used in many different institutions and contexts with different needs. It supports an extensive set of features that can be toggled on or off, based on the requirements of the deployment. These settings are known as “Feature Toggles” and most are documented in the Open edX documentation. The documentation contains a description, default value, and implementation type of each feature toggle available within the Open edX platform.

Feature toggles are implemented in a few different ways. The main three types are:

      1. Django Settings
      2. Toggles based on django-config-models
      3. Toggles based on Django Waffle

We already explained how to set Django Settings when using Tutor in the previous section. The toggles based on django-config-models or Django Waffle are easier to configure because the values are stored in the database and can be updated at runtime through Django Admin.

Some of the toggles (denoted as “CourseWaffleFlag” in the documentation) can also be overridden on the organization or course level by creating overrides in the WAFFLE_UTILS section of the Django Admin.

Be sure to confirm the impact of specific Waffle overrides in the documentation prior to implementing them.

Flags within edx-platform are documented on docs.openedx.org. Every flag defines its implementation type, for example Implementation: [‘DjangoSetting’] indicates a Django setting flag, and Implementation: [‘WaffleSwitch’] indicates a Waffle flag.

For example, the Waffle flag FEATURES[‘ENABLE_LTI_PROVIDER’] enables the capability of your site to be used as an LTI Provider to other systems and applications.

The Developer & Operator Release Notes for each Named Release contain a section that details all the new flags as well as the ones being deprecated. Flags that affect code defined in other repositories can be found by inspecting their code. The "Site Operators" section of the docs goes into depth on the overall explanation and how-tos for each of the cases we have in the Open edX system.
