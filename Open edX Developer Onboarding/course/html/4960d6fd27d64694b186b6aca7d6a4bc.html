Now let’s look at an important concept in Git called branches. Think of the work in a repository as a “trunk” (called either “master” or “main” in Open edX repos), and your work on a bug-fix or a feature as a branch coming out from the trunk. Ultimately, you will merge your branch back into the trunk (“master” or “main”). To learn more about branches, see this tutorial from Atlassian.

Branches enable multiple features/fixes to be developed at once, without impeding the progress of the trunk (“Master” branch). In this diagram, the “Master” branch has 4 commits. “branchA” branches off at the 1st commit and “branchB” branches off at the 3rd commit. As branches are developed, they often fall behind the trunk branch; when this happens, we follow a rebase workflow to get branches up to date. We’ll talk about rebasing in the next section. To create a new branch, you can run:

$ git branch<branchname>

This will create a branch but in order to switch to the branch you need to run:

$ git checkout <branchname>

Alternatively, we can use the checkout. command with the -b flag in order to both create a branch and switch to it at the same time:

$ git checkout -b <branchname>

Note: It simply switches to <branchname> if it already exists.

Only when there is no branch with that name does it create one. Now that you know how to create a branch, let’s look at how you can keep your branches and forks up to date.
