Tutor, by default, sets up and runs a copy of all services required by an Open edX instance (MySQL, MongoDB, Elasticsearch, Redis, object storage…).

For fault-tolerant production deployments, it is often preferred to use hosted solutions instead of running them yourself inside k8s. Hosted services usually come with built-in backup schemes and ability to use multiserver or even multi-region deployments for better reliability.

For example, a typical Open edX AWS production deployment will use RDS for MySQL, S3 for object storage, ElastiCache for Redis and OpenSearch Service as an equivalent of Elasticsearch, and perhaps a Mongo Atlas cluster for the MongoDB instance. You can view the architecture of the system in the Open edX documentation, although be aware that it may not always be up to date. As of May 2024, we are looking into removing both ElasticSearch and MongoDB; the best place to see what pieces of technology are deprecated or about to be deprecated is in the Deprecation section of the forums.

While you could in theory set all of these services and configure Tutor to use them manually, this is both time-consuming and error-prone, especially if you are hosting multiple instances or want to set up multiple environments (staging, production, etc.) for your instance.

If you are a DevOps engineer, you are probably already familiar with tools that let you automate infrastructure setup and make it reproducible. Popular tools for infrastructure provisioning are Terraform, OpenTofu, and Pulumi.

If you are managing your own Open edX instance, it is highly recommended to use one of these tools to set up your infrastructure.
