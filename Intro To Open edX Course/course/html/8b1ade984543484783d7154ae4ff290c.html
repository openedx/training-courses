A piece of software can be referred to as a product. Software products are successful when they provide value to the user of the product. Software products consist of the code that makes the software work and a user interface (what users of the software see and interact with). Think about YouTube - as a software product, it has a clear user interface that allows you to play and upload videos; there's also a huge invisible part of YouTube that you don't see, that manages the data of billions of videos and provides a backend infrastructure that allows it to stream any video, at any time, seamlessly. When we talk about a "software product", these invisible products are just as important as the parts revealed to us via the user interface.

The Open edX software product is comprehensive and wide-ranging. It includes many visible features, from checkbox problems to discussion forums to the Studio editor that allows course authors to author course content. There are also many invisible features such as the database management to hold all the course data and serve it up quickly and the integration code that enables course authors to pass an LTI unit into their course.

In this section, we'll be taking a look at what constitutes the Open edX product. Unlike YouTube, where all users are doing one or two things (consuming videos, maintaining a channel), the users of the Open edX software may be doing very different things. A college math instructor will likely create a very different type of course, with very different problem types, than someone writing a corporate compliance course on building safety standards. Understanding the different use cases that various people in our community have helps inform how we think about the project's product strategy.
